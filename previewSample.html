How to Deploy an API with Puppeteer on Render With And Without Docker: A Step-by-Step Guide

Francis Ihejirika

How To Deploy an API With Puppeteer On Render With And Without Docker

Introduction:

I previously wrote a guide on Deploying an API with Puppeteer on Google Cloud Run. I deployed the same codebase to Render as I often prefer to use Render as a development environment because I consider it the closest to what is obtainable on a local machine; this is to avoid the “it works on my machine, I’m not sure why it is not working on the server” type of situation.

There are two ways to get Puppeteer working on Render. We’ll discuss both methods and highlight common errors and workarounds.

Overview:

The deployed project used NodeJS/Express on the backend and React/Vite on the front end.

Below is a sample project structure showing the two directories in our repository — the API directory and the client directory. We are only concerned with the API directory as this is where Puppeteer is installed as a dependency. We will be deploying the API folder to Render.

What is Puppeteer?

Puppeteer is a library that helps you perform automated functions within your API. This could range from simple tasks such as generating screenshots to more complex tasks like running a script within a browser spun by the library, navigating to URLs to perform functions such as logging in as a user and returning some information to an external server, all without human intervention. The official page and documentation can be found here: https://pptr.dev/. According to the official website,

Puppeteer is a JavaScript library which provides a high-level API to control Chrome or Firefox over the DevTools Protocol or WebDriver BiDi. Puppeteer runs in the headless (no visible UI) by default (**Source, Puppeteer)

METHOD 1: Deploying To Render WITHOUT using a Dockerfile

When deploying to Render without using Docker, there are some configurations required to ensure your Puppeteer runs correctly. Here are the steps:

Step 1: Prepare your API directory

You do this by adding the Puppeteer configuration file to dynamically set the location where Chrome will be installed and spawned when Puppeteer attempts to run, afterwards, we’ll add the command to ensure Chrome is installed in the correct location.

  1. Add a Puppeteer configuration file named puppeteer.config.cjs at the root of your API directory. Note that the file extension should be exactly .cjs whether or not you are utilizing CommonJs or ES Modules in your API. The content of the file should be:
const { join } = require('path');

/**
 * @type {import("puppeteer").Configuration}
 */
module.exports = {
  // Changes the cache location for Puppeteer.
  cacheDirectory: join(__dirname, '.cache', 'puppeteer'),
};

2. Add a build command in your package.json file.

In your package.json file, add a “build”: “npm install && npx puppeteer browsers install chrome” command. This ensures that our project dependencies, as defined in the file are installed before Chrome is installed.

Step 2: Select the correct language, branch and directory on Render

When deploying the new web service to Render, you will be presented with a screen to connect your repository, select a unique name, language, branch and root directory for your project.

Connect the correct repository where your project files live using your preferred Git service, add a unique name for your service.

Select Node as the Language as we are deploying a NodeJS API, and given our project structure, we input /api as our root directory. This is also where we have our Puppeteer config file as well as our package.json file.

Step 3: Add the correct build and start commands, environment variables and deploy your service

Remember the “build” script command we added to our package.json file? This is where it comes in handy. For the Build Command on Render, you use your correct command, depending on what is setup in your package.json file.

Given our structure, for our Build Command we will input “npm install && npm run build” which will ensure our dependencies are installed and the “npx puppeteer browsers install chrome” command which is now a part of our build command as defined in our package.json file installs Chrome.

For the Start Command, we simply use npm start .

Afterwards, select the Instance Type, insert your environment variables and deploy your service. Your server should deploy correctly and your Puppeteer script work as expected.

Error Message If Puppeteer Configuration File and Correct Build Commands Are Missing

In the screenshot below, we can see two error messages:

  1. You did not perform an installation before running the script (e.g. ‘npx puppeteer browsers install chrome’

  2. Your cache path is incorrectly configured (which is: /opt/render/.cache/puppeteer)

These error messages will show up if we fail to include the Puppeteer configuration file or run the correct build command to install Chrome before deploying our Web API service.

METHOD 2: Deploying To Render using a Dockerfile

This is perhaps an easier way to ensure that our NodeJS app runs in any environment without encountering configuration errors, installation errors or having to manually specify an installation path within our codebase.

Step 1: Create a Dockerfile at the root of your directory

Create a file named “Dockerfile” in your API directory. Note that this is case sensitive, and as such, the ‘D’ should be capital, and without extensions — simply Dockerfile.

Add the following to your Dockerfile. In the code below, you can see we set the working directory for our app. This working directory may be needed depending on how you structure your Puppeteer script. An example is added below.

# Use the official Node.js image as the base image
FROM node:18-slim

# Set the working directory
WORKDIR /usr/src/app

# We don't need the standalone Chromium
ENV PUPPETEER_SKIP_CHROMIUM_DOWNLOAD true

# Install Google Chrome Stable and fonts
# Note: this installs the necessary libs to make the browser work with Puppeteer.
RUN apt-get update && apt-get install gnupg wget -y && \
    wget --quiet --output-document=- https://dl-ssl.google.com/linux/linux_signing_key.pub | gpg --dearmor > /etc/apt/trusted.gpg.d/google-archive.gpg && \
    sh -c 'echo "deb [arch=amd64] http://dl.google.com/linux/chrome/deb/ stable main" >> /etc/apt/sources.list.d/google.list' && \
    apt-get update && \
    apt-get install google-chrome-stable -y --no-install-recommends && \
    rm -rf /var/lib/apt/lists/*

# Verify that Chrome is installed at the expected location
RUN ls -alh /usr/bin/google-chrome-stable && \
    /usr/bin/google-chrome-stable --version

# Copy package.json and package-lock.json
COPY package*.json ./

# Install app dependencies including Puppeteer
RUN npm install

# Bundle app source code
COPY . .

# Expose the port on which your app will run
EXPOSE 8080

# Start the application
CMD ["npm", "start"]

The Dockerfile above ensures the following:

  1. Grabs the specified official Nodejs image

  2. Skipps downloading Chromium as we will be installing a stable version of Chrome in the next step.

  3. Installs a stable version of Chrome as well as other packages required by Puppeteer. Do note that the packages installed are for Debian Linux. If you prefer to have packages for a different distribution of Linux, you can look through the list in the link below to help:
    https://source.chromium.org/chromium/chromium/src/+/main:chrome/installer/linux/debian/dist_package_versions.json

  4. Adds a verification command to be certain that the installation is successful and exists in the correct location

  5. Copies the dependencies in the package.json file to the current working directory

  6. Installs the dependencies, including Puppeteer

  7. Copies the code from the current directory (API directory) into the current working directory as set in the Dockerfile, exposes the port and runs the start command

Step 2: Select Docker as Language and specify the Dockerfile path

After connecting the preferred repository, where our files are, we select Docker as the language from the language dropdown.

Our API is a NodeJs API in the actual sense but since we are containerizing the app using Docker, we simply select Docker, and Render takes care of the rest using the steps specified in the Dockerfile.

Next, we specify the root directory /api, and add the path to our Dockerfile. Since our Dockerfile is in the API root directory already specified above, we simply add a “.” to indicate this or leave blank.

After specifying the above, you simply select your instance type, include your environment variables and deploy your service.

Your server and Puppeteer scripts should run as intended.

Quick Note: Adding your installation path to your Puppeteer script

Depending on how your script is setup, you may be required to include the installation path to where your Chrome browser is installed.

Note that if you deploy on Render using method 1 above, where we directly included the path to the installation using the Puppeteer configuration file at the root of our directory, you will not be required to manually add this to your script.

On the contrary, if you make use of method 2, where we utilize Docker, you will be required to add the installation path. Make reference to the content of the Dockerfile where you specified the working directory, and simply deduce the installation path for Chrome.

Below is an example of my Puppeteer script using a Dockerfile where I inserted the path to my Chrome installation in my Dockerfile:

let browser;

        console.log('Attempting to automate admin login...');
        console.log('Launching Puppeteer...');

        let launchOptions = {
            headless: true,
            args: ['--no-sandbox', '--disable-setuid-sandbox', '--disable-dev-shm-usage', '--disable-accelerated-2d-canvas', '--disable-gpu'],
            executablePath: '/usr/bin/google-chrome-stable' // path to Google Chrome

        };

        browser = await puppeteer.launch(launchOptions);
        const page = await browser.newPage();

// continue with the rest of your script...

Conclusion:

Deploying an API using Puppeteer as one of the dependencies can be a little tricky but using the above methods, you should easily get a hang of it.

Puppeteer’s official website also has a comprehensive list of errors people may encounter when deploying to different environments such as on AWS Lambda, Heroku, Google Cloud Functions, Windows subsystem for Linux and a couple of others. You can check it out here: https://pptr.dev/troubleshooting.
